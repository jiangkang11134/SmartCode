# 严格审查触发场景

> 定义：什么情况下必须走"正则 + AST + 跨文件分析 + LLM 深度审查"全链路。
> 原则：宽松审查是默认通行证，严格审查是安检门——只对高风险路径开启。

## 二、按变更内容自动触发

### 2.1 正则层触发升级

宽松审查的正则发现以下模式时，**自动升级到严格**让 LLM 确认是否误报：

| 正则发现 | 风险 | 升级原因 |
|---------|------|---------|
| 疑似密钥 `api_key = "xxx"` | 密钥泄露 | 可能是测试 key 或配置文件，需 LLM 判断上下文 |
| 疑似 SQL 拼接 `f"SELECT {user_input}"` | SQL 注入 | 可能用了参数化查询但被正则误报 |
| 疑似 `eval()` 使用 | 代码注入 | 可能是合法的动态导入 |
| 硬编码密码 `password = "xxx"` | 凭据泄露 | 可能是测试密码 |
| 注释掉的代码 | 死代码 | 可能是有意保留的备选方案 |
| 文件包含 `base64` 长字符串 | 密钥/证书泄露 | 可能是正常的 base64 编码数据 |

升级逻辑：

```
宽松正则扫描
  │
  ├── 未发现可疑模式 → 输出宽松报告（结束）
  │
  └── 发现可疑模式 → 标记为 "需 LLM 确认"
       │
       ▼
       自动升级为严格审查
       LLM 读取上下文判断是真问题还是误报
       │
       ├── 真问题 → 输出严格报告（含 severity 和建议修复方案）
       │
       └── 误报 → 在报告中标注 "已确认为误报"
```

### 2.2 AST 层触发升级

| AST 发现 | 风险 | 升级原因 |
|---------|------|---------|
| 循环复杂度 > 15 | 可维护性差 | LLM 需要分析逻辑是否可简化 |
| 函数参数 > 8 个 | 设计不合理 | LLM 需要判断是否该用配置对象 |
| 存在多条返回路径 | 逻辑混乱 | LLM 需要确认是否有遗漏的 return |
| 异常被吞掉 `except: pass` | 错误处理缺失 | LLM 需要判断是否故意为之 |
| 深层嵌套 > 5 层 | 可读性差 | LLM 建议重构方案 |
| 全局变量修改 | 副作用的潜在风险 | LLM 判断是否影响线程安全 |

### 2.3 diff 特征触发

通过 `git diff` 分析变更的特征，决定是否升级：

```python
DIFF_TRIGGERS = {
    "changed_lines > 200":            # 单次变更超过 200 行
        "大规模变更，潜在影响范围大",
    "touched_files > 10":             # 跨 10 个以上文件
        "跨文件变更，需要分析调用链",
    "deleted_lines > 100":            # 删除超过 100 行
        "大量删除，确认是否误删",
    "public_api_changed":             # 修改了 public 接口签名
        "API 变更，需确认是否兼容",
    "interface_protocol_changed":     # 修改了 Protocol/ABC
        "接口协议变更，影响所有实现方",
    "test_files_included":            # 同时修改了测试文件
        "有测试变更，需要审查测试本身的质量",
    "test_files_not_included":        # 改代码但没有改测试
        "缺少测试，需要判断是否需要补充",
    "config_files_changed":           # 修改了配置
        "配置变更，需关注对环境的影响",
    "framework_version_changed":      # 改动了依赖版本
        "依赖升级，需要审查 breaking changes",
}
```

## 三、按场景自动触发

### 3.1 PR / MR 审查

PR 进来的代码**不是自己写的**，不存在"过于自信"的问题，**强制严格**：

```bash
minicode-py --review-pr owner/repo 42
# → 自动走严格审查，无宽松选项
```

因为 PR 审查的场景假设是：**代码是别人的，你作为 reviewer 必须全面检查。**

### 3.2 安全审查专项

```bash
minicode-py --review security
# → 强制严格，且只跑 security 分类的规则
```

场景：团队要求上线前做安全审计。此时只看安全问题，不看代码风格。

### 3.3 准入审查（Gate Review）

CI 流水线中整合，阻止有严重问题的代码合入主分支：

```bash
# CI 中
minicode-headless --review gate --diff-to main
# → 强制严格，发现 critical 级别的问题 → exit code 1
```

### 3.4 上线的最后一次 Review

```bash
minicode-py --review strict --export sarif
# → 完整全量审查，输出 SARIF 格式给 GitHub 或 VSCode 使用
```

## 四、按历史模式自动触发

### 4.1 历史问题文件

`review_memory.py` 存储了每次 review 的历史。如果某个文件**过去 3 次 review 中 2 次以上有问题**，后续每次审它都自动严格：

```python
def should_force_strict(file_path: str, store: ReviewMemoryStore) -> bool:
    findings = store.find_by_file(file_path)
    recent = [f for f in findings if is_within_days(f.created_at, 90)]
    if len(recent) < 3:
        return False
    problematic = sum(1 for f in recent if f.status in (
        ReviewStatus.OPEN, ReviewStatus.ACKNOWLEDGED
    ))
    return problematic / len(recent) >= 0.6  # 60% 以上有问题
```

### 4.2 频繁变动的文件

Git 历史显示频繁修改的文件（hot spot）——出 bug 的概率更高：

```python
def should_force_strict_by_git_history(file_path: str) -> bool:
    # 最近 90 天修改次数 > 10，或修改人数 > 5
    # → 代码锅味重，自动严格
```

### 4.3 新人的代码

如果 git blame 显示改动的 author 是近期才加入项目的（可通过 `.git` 历史或手动配置），自动严格审查。新人可能不熟悉项目规范和代码风格，**不是信任问题，是信息差问题。**

```python
def should_force_strict_by_author(author: str, known_authors: set[str]) -> bool:
    return author not in known_authors
```

## 五、总结：触发优先级矩阵

```python
def decide_review_mode(file_path: str, diff_stat: dict, author: str,
                       store: ReviewMemoryStore, git_history: list) -> str:
    """
    返回: "loose" / "normal" / "strict"
    """
    # 优先级 1: 安全文件路径
    if any(fnmatch.fnmatch(file_path, pattern) for pattern in STRICT_PATHS):
        return "strict"

    # 优先级 2: diff 特征
    for trigger, reason in DIFF_TRIGGERS.items():
        if check_diff_trigger(diff_stat, trigger):
            return "strict"

    # 优先级 3: 历史模式
    if should_force_strict_by_history(file_path, store):
        return "strict"

    # 优先级 4: 新人代码
    if should_force_strict_by_author(author, known_authors):
        return "strict"

    # 优先级 5: 复杂度超限（AST 分析后决定）
    # 实际在扫描过程中动态升级
    return "normal"  # 普通走 normal
```

```
触发顺序：

用户发起审查
    │
    ├── 路径匹配安全文件？ → 直接 strict，跳过后续判断
    │
    ├── 是 PR 模式？        → 直接 strict
    │
    ├── 是安全审查专项？     → 直接 strict
    │
    ├── diff 特征触发？      → strict，标注触发原因
    │
    ├── 历史模式触发？       → strict，标注触发原因
    │
    ├── 新人代码？           → strict，标注"新 contributor"
    │
    └── 以上都不是？
            │
            ├── normal 审查
            │   ├── 正则发现可疑模式？ → 自动升级 strict
            │   ├── AST 发现复杂度超限？ → 自动升级 strict
            │   └── 都正常 → 输出 normal 报告
            │
            └── strict 审查 → 全链路
```

## 六、配置化（`review_rules_db.py`）

所有触发规则应该是**可配置的**，而不是硬编码：

```python
# review_rules_db.py

DEFAULT_STRICT_CONFIG = {
    "file_paths": STRICT_PATHS,                     # 路径匹配
    "diff_triggers": DIFF_TRIGGERS,                 # diff 特征
    "regex_escalations": {                          # 正则升级到 LLM 确认
        "api_key_pattern": "密钥/Token 疑似泄露",
        "sql_concat_pattern": "疑似 SQL 注入",
        "eval_pattern": "疑似不安全 eval",
        "password_hardcode": "疑似硬编码密码",
        "commented_code": "疑似遗留死代码",
        "base64_secret": "疑似 Base64 编码的密钥",
    },
    "ast_escalations": {                            # AST 升级到 LLM 确认
        "high_complexity": { "threshold": 15, "message": "循环复杂度过高" },
        "deep_nesting": { "threshold": 5, "message": "嵌套过深" },
        "too_many_params": { "threshold": 8, "message": "参数过多" },
    },
    "complexity_thresholds": STRICT_COMPLEXITY_THRESHOLDS,
    "history": {
        "force_strict_if_problem_rate": 0.6,        # 历史问题率 > 60%
        "history_lookback_days": 90,                 # 回顾 90 天
        "min_samples": 3,                            # 至少 3 条历史记录才启用
    },
    "git": {
        "force_strict_if_frequent_edits": 10,       # 90 天修改 > 10 次
        "force_strict_if_many_authors": 5,           # 涉及作者 > 5 人
        "newcomer_lookback_days": 30,                # 新人定义：加入 < 30 天
    },
}
```
